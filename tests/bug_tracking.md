## 缺陷管理流程

```
测试发现 Bug
    ↓
提交 Bug 报告（缺陷登记表）
    ↓
开发负责人确认（24 小时内）
    ↓
分配给开发人员
    ↓
开发修复 + 自测
    ↓
更新 Bug 状态为"待验证"
    ↓
测试回归验证
    ↓
验证通过 → 关闭
验证失败 → 重新打开
```

## 缺陷分级标准

### 严重程度

| 级别 | 名称 | 说明 | 响应时间 |
|------|------|------|----------|
| P0 | 阻塞性 | 系统崩溃、核心功能不可用 | 立即修复 |
| P1 | 严重 | 主要功能失效、数据错误 | 24 小时内 |
| P2 | 一般 | 次要功能问题、UI 错误 | 3 天内 |
| P3 | 轻微 | 建议优化、文案问题 | 下个版本 |

### 优先级

| 级别 | 名称 | 说明 |
|------|------|------|
| P0 | 紧急 | 影响上线，必须立即修复 |
| P1 | 高 | 影响核心功能，优先修复 |
| P2 | 中 | 正常排期修复 |
| P3 | 低 | 有空时修复 |

## 缺陷流转规则

### 状态定义

| 状态 | 说明 | 责任人 |
|------|------|--------|
| 新建 | 测试提交 Bug | 测试工程师 |
| 已确认 | 开发确认 Bug 存在 | 开发负责人 |
| 修复中 | 开发正在修复 | 开发人员 |
| 待验证 | 开发修复完成，等待测试验证 | 开发人员 |
| 已关闭 | 测试验证通过 | 测试工程师 |
| 重新打开 | 测试验证失败，退回开发 | 测试工程师 |

### 流转时限

| 流转 | 时限 | 超时处理 |
|------|------|----------|
| 新建 → 已确认 | 24 小时 | 自动升级 |
| 已确认 → 修复中 | 48 小时 | 自动升级 |
| 修复中 → 待验证 | 按 Bug 级别 | 自动升级 |
| 待验证 → 已关闭 | 24 小时 | 自动关闭 |
